This Solution Design Document (SDD) defines the complete technical architecture, integration design, data model, and implementation approach for a native mobile application available on Android and iOS. The application leverages Salesforce Nonprofit Success Pack (NPSP) as the single source of truth for constituent management, Firebase as the real-time backend and notification layer, and PhonePe as the primary UPI payment gateway. The Cloud Functions environment holds a private key (stored in Firebase Secret Manager) and mints signed JWTs to exchange for Salesforce access tokens on-demand. Step Detail 1. Key Generation RSA 2048-bit key pair generated. Public key uploaded to Salesforce Connected App. Private key stored in Firebase Secret Manager as secret: sf-private-key. 2. JWT Minting Cloud Function builds JWT: { iss : consumerKey , sub: sfUsername , aud : https://login.salesforce.com, exp: now+ 300 } . Signed with RS256. 3. Token Exchange POST to https://login.salesforce.com/services/oauth2/token with grant_type = urn:ietf : params:oauth :grant- type:jwt -bearer . Returns access_token + instance_url . 4. Token Caching Access token cached in Firebase in-memory for 4 minutes (expires at 5). Refreshed automatically before expiry. 5. API Calls All Salesforce REST calls use Authorization: Bearer { access_token } header against instance_url . 6.1.2 Contact Sync Trigger: Firebase Auth onUserCreate Logic: 1. Search SF: GET /services/data/v59.0/ query?q =SELECT Id FROM Contact WHERE Phone = '{phone}' 2. If found: PATCH Contact with Mobile_App_ID__c = uid , App_Registration_Date__c = today 3. If not found: POST new Contact with all available fields 4. Store returned Contact.Id in Firestore user doc as sfContactId 6.1.3 Payment → Opportunity Sync Trigger: PhonePe webhook SUCCESS event Payload built by Cloud Function: POST /services/data/v59.0/ sobjects /Opportunity/ { Name: "{category} — { displayName } — {Month Year}", Amount: {amount}, CloseDate : "{today}", StageName : "Closed Won", Type: "Donation", AccountId : "{ sfAccountId from Contact}", PhonePe_Transaction_ID__c : "{ merchantTransactionId }", Giving_Category__c : "{category}", Source__c : "Mobile App" } Then POST OpportunityContactRole : { OpportunityId , ContactId , Role: "Donor", IsPrimary : true } 6.2 PhonePe Payment Gateway Integration 6.2.1 Payment Initiation Flow Step Description 1. User taps Pay App calls createGivingOrder Cloud Function (Callable) with amount and category. 2. Order creation Cloud Function POSTs to PhonePe : https://api.phonepe.com/apis/hermes/pg/v1/pay with payload: { merchantId , merchantTransactionId , amount (paise), redirectUrl , callbackUrl , paymentInstrument : { type : PAY_ PAGE } }. Payload base64-encoded, checksum = SHA256(base64payload + / pg /v1/pay + saltKey ) + "###" + saltIndex . 3. Order stored Pending transaction written to Firestore givingTransactions with status: pending. 4. redirectUrl returned Cloud Function returns PhonePe redirectUrl to mobile app. 5. PhonePe SDK launched Mobile app calls PhonePe SDK with redirectUrl . Native payment sheet shown. 6. User pays User selects UPI app / card. Completes payment on PhonePe sheet. 7. Redirect back PhonePe redirects to app via universal link (iOS) / App Link (Android). 6.2.2 Webhook Fulfilment Flow PhonePe POSTs to: https://{region}-{projectId}.cloudfunctions.net/handlePhonePeWebhook Header: X-VERIFY: {checksum} Body: { "response": "{base64EncodedPayload}" } Verification: checksum_computed = SHA256(response_base64 + saltKey ) + "###" + saltIndex if checksum_computed !== X-VERIFY → return HTTP 400, log security alert On verification pass: 1. Decode base64 → parse JSON → check code === "PAYMENT_SUCCESS" 2. Check Firestore : if txId already status:success → return 200 ( idempotent) 3. Update Firestore : status:success , phonepeTransactionId , completedAt 4. Call createSalesforceOpportunity ( ) 5. Call generatePDFReceipt ( ) → upload to Storage → send FCM to user 6. Return HTTP 200 6.3 Firebase FCM Notification Design Notification Type Trigger Target Payload Daily promise Cloud Scheduler 7:00 AM IST All users with promise notif enabled title: Today's Promise, body: first 80 chars of verse, data: {screen: promise, date} New daily video Firestore onCreate ( dailyVideos ) All users with video notif enabled title: Today's 1-min message is ready!, body: video title, data: {screen: video, date} Event reminder 24hr Cloud Scheduler 9:00 AM IST daily RSVPd users for tomorrow's event title: Reminder: {event title}, body: Tomorrow at {time} — {venue} Event reminder 1hr Cloud Scheduler — hourly check RSVPd users for events in next 60 min title: Starting soon: {event}, body: In 1 hour at {venue} Prayer response Cloud Function on pastor reply Specific user (by uid / FCM token) title: Pastor has responded to your prayer, body: Tap to read the message Prayer prayed Cloud Scheduler 6:00 PM IST Users whose requests got prays today title: {N} people prayed for you today, body: {request preview} Payment success PhonePe webhook success Specific user title: Giving confirmed — ₹{amount}, body: Thank you for your {category}. Token verified server-side automatically. Admin Portal → Firebase Firebase Auth JWT (email/password) Custom Claims: { role : "admin" | "pastor" | "finance " } . Claims checked in Firestore Rules and Cloud Functions. Cloud Functions → Salesforce OAuth2 JWT Bearer Flow Private key in Firebase Secret Manager. Token cached 4 min. No user credentials stored. PhonePe Webhook → Functions SHA256 checksum (HMAC-like) X-VERIFY header verified on every incoming webhook. Invalid checksum → HTTP 400, security log entry. Functions → PhonePe API Key + Salt saltKey stored in Firebase Secret Manager. Never exposed to client. 7.2 Firestore Security Rules Key rules (simplified): users/{ uid }: read: request.auth.uid == uid || isAdmin () write: request.auth.uid == uid // user edits own profile only prayerRequests /{ reqId }: read (public): resource.data.type == "public" &amp;&amp; resource.data.status == "approved" read (private): isAdmin () || isOwner () create: request.auth != null update (testimony): isOwner () &amp;&amp; request.resource.data.status == "testimony" update (approve/reject): isAdmin () givingTransactions /{ txId }: read: isOwner () // user can only read own transactions write: false // all writes via Cloud Functions only (admin SDK bypasses rules) function isAdmin () { return request.auth.token.role == "admin" || request.auth.token.role == "pastor"; } function isOwner () { return request.auth.uid == resource.data.uid ; } 7.3 Data Privacy &amp; Compliance Requirement Implementation Payment data security (PCI-DSS) No card data stored on church servers or in Firestore . All card/UPI data handled exclusively by PhonePe PG. Church receives only a transaction ID. Personal data minimisation Only name, phone, city, and cell group collected on registration. Photo optional. No biometric data. Private prayers Private prayer requests stored with type=private in Firestore . Security rules ensure only the requester and admin can read them. Not shown in public prayer wall. Data retention Giving transactions retained indefinitely for tax records. Prayer requests: soft-deleted after 1 year if status=rejected. User accounts: deleted on request. Secrets management All API keys, Salesforce private key, PhonePe saltKey stored in Firebase Secret Manager. Never in source code or client bundles. HTTPS everywhere All Cloud Function endpoints over HTTPS only. Firebase Hosting enforces HTTPS. Admin portal uses Firebase Hosting with HSTS. §8 Module Functional Design Screen-level and feature-level specifications 8.1 Daily Promise Module Feature Functional Requirement Technical Implementation Verse display Show today's verse, reference, devotional note, and pastor name on a branded navy card Firestore real-time listener on dailyPromises /{today}.
